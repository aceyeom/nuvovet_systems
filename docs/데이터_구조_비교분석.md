# 데이터 구조 비교: `frontend/src` 요구사항 vs `backend/db/scema.jsonl`

## 1) 결론 요약
- 현재 `scema.jsonl`에는 임상 텍스트와 일부 구조화 값이 있어 **기초 매핑은 가능**합니다.
- 하지만 프론트엔드(`DrugInput`, `durEngine`, `OrganLoadIndicator`, `ConfidenceProvenance`)가 바로 쓰는 필드 기준으로는 **핵심 런타임 필드가 다수 누락**되어 있습니다.
- 즉, "데이터가 완전히 없음"보다는, **엔진 입력용 정규 필드(불리언/enum/숫자)로 미정규화**된 상태가 가장 큰 갭입니다.

## 2) 비교 기준 (실제 프론트 사용 필드)

아래 파일에서 실제 접근하는 필드를 기준으로 비교했습니다.
- `frontend/src/utils/durEngine.js`
- `frontend/src/components/DrugInput.jsx`
- `frontend/src/components/OrganLoadIndicator.jsx`
- `frontend/src/components/ConfidenceProvenance.jsx`
- `frontend/src/components/DrugTimeline.jsx`

백엔드 비교 대상:
- `backend/db/scema.jsonl`

## 3) 필드 매핑 결과

상태 정의:
- `직접 매칭`: 프론트가 바로 사용 가능한 형태.
- `가공 필요`: 원천은 있으나 형태/타입 변환 필요.
- `누락`: `scema.jsonl`에서 사실상 없음.

| 프론트 필요 필드 | `scema.jsonl` 소스 | 상태 | 비고 |
|---|---|---|---|
| `name` | `drug_identity.name_en` | 가공 필요 | 영문명 매핑 필요 |
| `nameKr` | `drug_identity.name_ko` | 가공 필요 | 키 이름만 변환 |
| `class` | `drug_identity.class` | 직접 매칭 | |
| `activeSubstance` | (명시 없음, `name_en`로 대체 가능) | 가공 필요 | 동의어/성분 분리 규칙 필요 |
| `id` (stable key) | 없음 | 누락 | slug/canonical id 생성 필요 |
| `source` (`kr_vet` 등) | 없음 | 누락 | 출처 enum 필드 필요 |
| `route` (대표 투여경로) | `dosage_and_kinetics.*.dosage_list[].route` | 가공 필요 | 다중 route -> 단일 기본 route 결정 규칙 필요 |
| `freq` | `dosage_and_kinetics.*.dosage_list[].frequency` | 가공 필요 | `q8h` -> `TID/BID/SID` 정규화 필요 |
| `defaultDose.dog/cat` | `dosage_and_kinetics.*.dosage_list[].value` | 가공 필요 | 문자열 범위(`"4 - 10"`) 파싱 필요 |
| `speciesNotes.dog/cat` | `section_1_2_10`, `organ_systems_impact`, `precautions` | 가공 필요 | 현재 자유서술 텍스트 |
| `contraindications[]` | `precautions.*.evidence` | 가공 필요 | 문장 -> 리스트 추출 필요 |
| `mdr1Sensitive` | `genetic_sensitivity.has_genetic_risk`, `affected_breeds` | 가공 필요 | MDR1 특이 플래그 직접 없음 |
| `narrowTherapeuticIndex` | 없음 | 누락 | NTI 직접 플래그 필요 |
| `serotoninSyndromeRisk` | 없음 | 누락 | 엔진 룰용 boolean 필요 |
| `electrolyteEffect` (`k_depleting` 등) | 일부 서술(`drug_interactions` evidence) | 누락 | 현재 기계판독 가능한 enum 없음 |
| `cypProfile.substrate[]` | 없음 | 누락 | CYP substrate/inhibitor/inducer 배열 필요 |
| `cypProfile.inhibitor[]` | 없음 | 누락 | 동상 |
| `cypProfile.inducer[]` | 없음 | 누락 | 동상 |
| `riskFlags.qtProlongation` | `dosage_and_kinetics.*.qt_prolongation` | 가공 필요 | bool -> `none/low/moderate/high` enum 변환 필요 |
| `riskFlags.bleedingRisk` | 없음 | 누락 | |
| `riskFlags.hepatotoxic` | 없음 | 누락 | |
| `riskFlags.nephrotoxic` | 없음 | 누락 | |
| `renalElimination` (0~1) | 없음 | 누락 | |
| `hepaticElimination` (0~1) | 없음 | 누락 | |
| `pk.halfLife` (numeric) | `dosage_and_kinetics.*.half_life` (`"Data variable"`) | 누락 | 숫자형 PK 부재 |
| `pk.timeToPeak` | `timing_profile.peak_min` | 가공 필요 | 분 -> 시간 변환 필요 |
| `pk.bioavailability` | 없음 | 누락 | |
| `pk.proteinBinding` | 없음 | 누락 | |
| `pk.primaryElimination` | 없음 | 누락 | |
| `offLabelNote/foreignNote/unknownNote` | 없음 | 누락 | UI 설명용 필드 |

## 4) "진짜로 빠진 것"만 묶은 목록

아래는 단순 키 변환으로 해결되지 않고, `scema.jsonl` 기준으로 새로 생성/보강이 필요한 항목입니다.

1. 엔진 핵심 룰 입력
- `cypProfile.substrate[]`
- `cypProfile.inhibitor[]`
- `cypProfile.inducer[]`
- `serotoninSyndromeRisk`
- `electrolyteEffect`
- `narrowTherapeuticIndex`

2. 장기부담/독성 정량 필드
- `renalElimination`
- `hepaticElimination`
- `riskFlags.bleedingRisk`
- `riskFlags.nephrotoxic`
- `riskFlags.hepatotoxic`

3. PK 시각화 필수 숫자
- `pk.halfLife` (numeric)
- `pk.bioavailability`
- `pk.proteinBinding`
- `pk.primaryElimination`

4. 프론트 식별/출처 메타
- `id` (stable/canonical)
- `source` (`kr_vet`, `human_offlabel`, `foreign`, `unknown`)

## 6) 최종 판단

- 질문: "frontend src에서 필요한 데이터가 `backend/db/scema.jsonl`에 맞아?"
- 답: **부분적으로만 맞습니다.**
  - 표시용 기본 정보(`name`, `nameKr`, `class`)는 매핑 가능.
  - DUR 룰 엔진과 장기부담/PK 시각화에 필요한 핵심 필드는 다수가 누락 또는 비정규 형태입니다.

따라서 현재 상태로는 직접 연결이 어렵고, 변환 레이어(ETL/adapter) 없이는 프론트 기능을 안정적으로 충족하기 어렵습니다.
